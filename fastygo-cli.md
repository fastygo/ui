как в Go реализовать CLI в shadcn стиле чтобы клонировать компоненты в приложение напрямую из github без необходиости ставить всю библитоку

`fastygo add hero button`

------------------------

В Go это делают **отдельной маленькой CLI** (часто `go install github.com/fastygo/cli/cmd/fastygo@latest`), по духу как shadcn: **не ставите всю библиотеку**, а **скачиваете/копируете только выбранные файлы** в дерево проекта и дальше они живут как **ваш код**.

### Что именно повторяет идею shadcn

1. **Реестр (manifest)** — JSON/YAML: имя компонента → список файлов, опционально зависимости.  
   Пример логики: `hero` тянет `button`, `utils`; `button` тянет только `utils`.

2. **Источник файлов** (любой из вариантов):
   - **Статическая вставка в бинарник** — `//go:embed registry/...` в CLI: без сети, версия CLI = версия реестра.
   - **GitHub raw по тегу** — `GET https://raw.githubusercontent.com/fastygo/ui/v1.2.3/blocks/hero/hero.templ` по путям из manifest.
   - **Один «bundle» JSON** как у shadcn registry — в нём уже `content` файлов (удобно для атомарных релизов).

3. **Команда** — разбор аргументов: `fastygo add hero button` → два компонента, объединить зависимости, топологическая сортировка, без дубликатов.

4. **Запись на диск** — целевая папка из флага или конфига, например `fastygo.json`: `"componentsDir": "internal/ui"`.

5. **Подстановка путей** — как `@/components` у shadcn: при копировании заменить плейсхолдеры на **реальный module path** из `go.mod` приложения (`github.com/you/app/internal/ui/...`), чтобы импорты собирались.

6. **После копирования** — при необходимости вызвать `go tool templ generate` / `gofmt` (опционально).

### Минимальный каркас CLI в Go

- Парсинг: `cobra` или `flag` + подкоманда `add`.
- HTTP: `net/http` + контекст + таймауты для загрузки с GitHub.
- Файлы: `os.MkdirAll`, `os.WriteFile`, проверка «файл уже есть — пропустить / перезаписать / спросить».
- Реестр: один `registry.json` в репозитории `ui` или отдельном `ui-registry`, CLI знает URL или embed.

### Важные отличия от npm-shadcn

- В Go **нет** «одного пакета на компонент» в том же смысле: после `add` это **ваши** `.go`/`.templ` с импортами на `ui8kit` или на ваш `fastygo/ui` — зависимости всё равно тянет `go mod`, но **не нужно** тянуть весь монорепозиторий компонентов как библиотеку, если вы копируете только нужные файлы.
- Имеет смысл в manifest явно указывать **минимальные go-модули**, которые должны быть в `go.mod` (например `github.com/fastygo/ui8kit`), и команда `add` может печатать: «выполните `go get ...` если ещё нет».

### Итог

**Да:** `fastygo add hero button` = CLI читает реестр → разрешает зависимости → качает/вставляет файлы → подставляет module path → пишет в `internal/ui` (или куда настроите).  
**Реализация в Go** — обычное приложение с подкомандой + реестр + копирование/шаблонизация; стиль shadcn — это в первую очередь **реестр + копирование исходников**, а не `go get` всей библиотеки.
